# Vendor Selection Brief — SSO & User Database
**Project:** CazéTV World Cup 2026 Predictor (Bolão)
**Date:** April 2026
**Status:** 🔴 Urgent — Critical blocker for LiveLike integration

---

## 1. Project Context

CazéTV is building a **digital prediction game (bolão) for the 2026 FIFA World Cup**, a fan engagement platform where users make match predictions, earn points, compete in leagues, and receive prizes. More than **15 million** users are expected to create accounts during the 2026 World Cup.

The engagement platform will be built on top of LiveLike. **LiveLike does not store identity data (PII).** It operates on a federated identity model: it receives an authentication token generated by our own identity system (SSO), creates an internal mapping (LiveLike ID ↔ CazéTV ID), and only stores gameplay data. **All user identity — registration, login, authentication — is exactly what we are looking to build with a vendor.**

---

## 2. The Challenge

We need to hire a vendor who takes full ownership and responsibility for:

1. **Designing and implementing** CazéTV's digital identity system and user database
2. **Operating and maintaining** this infrastructure throughout the entire project lifecycle (World Cup 2026 and beyond)
3. **Guaranteeing SLA for availability and scalability**, especially during extreme traffic spikes

We are talking about:

- **Mass B2C audience** — Brazilian football fans with no technical background
- **Unpredictable registration spikes** — expectation of **millions of account creations within minutes** during commercial announcements and platform launch
- **Aggressive delivery window** — the database must be live and integrated with LiveLike by end of April 2026
- **Future extensibility** — the architecture must support the eventual addition of a sponsor as a second identity provider, without full refactoring

---

## 3. Expected Vendor Scope

### 3.1 Implementation (Phase 1 — Critical)

| Deliverable | Description |
|---|---|
| **User database** | Scalable database to store CazéTV user identity, PII, and credentials |
| **Registration flow** | Account creation via email/password or social login (Google, Apple, Facebook) |
| **Authentication & session** | Issuance of JWT/OIDC token compatible with LiveLike's expected standard |
| **SSO endpoint** | Authentication endpoint called by LiveLike to validate the user and receive the token |
| **LiveLike handshake** | Integration and certification of the full flow: CazéTV Auth → LiveLike ID mapping |
| **Peak traffic management** | Auto-scaling strategy and warm-up plan to support massive simultaneous account creation → What is the peak req/sec the system can handle? |

### 3.2 Ongoing Operations (Phase 2 — Sustained)

| Deliverable | Description |
|---|---|
| **24/7 monitoring** | System health dashboards with proactive alerting |
| **Uptime SLA** | Minimum 99.9% availability — especially during match days |
| **Incident response** | Maximum 5-min response time for P1 incidents (auth down = platform down) |
| **Security updates** | Patches and preventive maintenance included in the contract |
| **Full documentation** | All architecture documented and handed over to CazéTV (no lock-in) |

---

## 4. Minimum Technical Requirements

> We are **not tied** to any specific solution (Auth0, Firebase, Cognito, etc.). What matters is the outcome and contractual accountability.

- Native support for **OIDC / OAuth 2.0 / JWT** — mandatory for LiveLike integration
- Architecture hosted on **public cloud** with auto-scaling configured
- Proven capacity to handle **peaks of 100k+ req/min** for authentication
- Compliance with **Brazil's LGPD** — PII stored in Brazil or with a proper data processing agreement
- Extensible architecture for **future identity federation** (e.g., Sponsor as a secondary IdP)
- Delivery of a **functional database by end of April 2026**

## 5. What We Expect in the Proposal

We ask vendors to include in their response:

1. **Proposed architecture** — technical overview of the identity solution
2. **Detailed timeline** — from kick-off to LiveLike integration
3. **Reference cases** — B2C projects with similar traffic spikes
4. **SLA model** — uptime, incident response times, maintenance scope
5. **Cost breakdown** — implementation (phase 1) and monthly operations (phase 2), separately
6. **Extensibility plan** — how they would support adding a secondary IdP (sponsor) without full refactoring
7. **Robustness evidence** — proof of solution robustness. What is the peak requests/sec the system can sustain?
